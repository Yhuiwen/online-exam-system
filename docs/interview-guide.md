# Java 后端实习面试问题精修版：在线考试与智能题库管理系统

> 适用场景：Java 后端实习、JavaWeb 课程设计答辩、简历项目追问。回答要围绕真实实现展开，不主动夸大为高并发、分布式或企业级微服务项目。

## 1. 30 秒项目介绍

这个项目是我基于 Spring Boot 3 和 Vue 3 完成的前后端分离在线考试与智能题库管理系统，面向管理员、教师和学生三类用户。后端主要负责登录认证、角色权限控制、题库管理、手动和自动组卷、在线答题、客观题自动判分、主观题人工批改、错题记录、成绩统计和考试异常行为记录。项目中我重点实践了 Spring Security + JWT、MyBatis-Plus、MySQL 表设计、Excel 批量导入导出、前后端接口联调、Docker 化运行，以及 AI 出题和轻量级 RAG 知识库答疑。

## 2. 2 分钟项目介绍

我做这个项目是因为在线考试系统业务链条比较完整，既有用户权限，也有题库、考试、答题、判分、批改和统计，适合作为 Java 后端综合项目。

系统分为管理员、教师、学生三类角色。管理员主要管理用户和基础数据；教师负责维护题库、创建考试、组卷、发布考试、批改主观题和查看成绩统计；学生负责参加考试、提交答案、查看成绩和错题。后端采用 Spring Boot 分层开发，使用 Spring Security + JWT 做认证授权，使用 MyBatis-Plus 操作 MySQL，前端使用 Vue3 和 Element Plus 完成页面和接口联调。

最能体现后端能力的地方有四个：第一是权限控制，不仅做角色鉴权，还在 Service 层校验资源归属；第二是考试业务状态流转，发布后试卷锁定，学生答题后进入提交或待批改状态；第三是 Excel 题库导入，包含模板、字段校验、重复题检测和错误行提示；第四是数据库设计，把考试、题目、学生答卷、单题答案、错题和异常行为分表处理，方便后续统计和扩展。

## 4. 数据库设计快速讲法

可以这样讲：

> 数据库核心是围绕考试闭环设计的。`sys_user` 管用户和角色，`course` 管课程，`question` 管题库，`exam` 管考试，`exam_question` 是考试和题目的中间表，保存题目顺序和考试分值。学生参加考试时生成 `student_exam`，每道题答案写入 `student_answer`。如果是客观题可以自动判分，如果是主观题则进入待批改状态。答错的题会沉淀到 `wrong_question`，考试异常行为记录到 `exam_violation`。这种设计把题库、试卷、答卷、单题答案、错题和异常行为拆开，后续统计、批改和扩展都比较清晰。

更详细的字段和约束见：[数据库设计说明](database-design.md)。

## 5. 高频面试问题与回答

### Q1：JWT 登录流程是什么？

用户提交用户名和密码，后端先查用户，再用 BCrypt 校验密码。校验通过后生成 JWT 返回给前端，前端保存 token，后续请求在请求头携带。后端过滤器解析 token，校验用户是否存在、是否禁用，再把认证信息放入 Spring Security 上下文。

### Q2：为什么用了 JWT 还要查数据库？

JWT 只能证明这个 token 曾经被签发过，但用户可能后来被禁用，角色也可能被管理员修改。如果完全相信 token 中的角色，会出现旧 token 继续访问的问题。所以后端解析 token 后仍然查询数据库，以数据库中的 `status` 和 `role` 为准。

### Q3：Spring Security 在项目里起什么作用？

它主要负责认证和授权。认证方面，通过过滤器解析 JWT，建立登录态；授权方面，通过路径配置和注解限制不同角色能访问的接口。更细的资源归属，比如教师是否能修改某道题，则放到 Service 层校验。

### Q4：三角色权限如何设计？

用户表里有 `role` 字段，主要包括管理员、教师和学生。管理员可以管理全局数据；教师可以维护课程、题库、考试和批改；学生只能参加考试和查看自己的结果。角色鉴权解决“能不能进某类接口”，资源校验解决“能不能操作这条数据”。

### Q5：如何防止教师修改其他教师创建的题目？

题目表里有 `create_user_id` 字段。教师编辑或删除题目时，Service 层会查询题目创建者，判断是否等于当前登录教师 ID；管理员可以放行。这样即使前端隐藏了按钮，后端也不会只依赖前端控制。

### Q6：教师为什么能用同课程其他老师的题来组卷？

组卷是只读引用题目，不一定等于拥有题目。题目编辑删除要校验创建者，而组卷可以按课程维度筛选题目并写入 `exam_question`。这能提高题库复用率，同时不破坏题目所有权。

### Q7：学生答题时如何避免看到答案？

后端给学生返回试卷题目时，只返回题干、选项、分值等必要信息，不返回 `answer` 和 `analysis`。答案和解析只在教师端、批改端或考试结束后的特定场景展示。核心原则是敏感字段不能只靠前端隐藏，后端返回时就要控制。

### Q8：为什么考试和题目之间要用 `exam_question` 中间表？

因为一个考试有多道题，一道题也可能被多个考试复用，这是多对多关系。同时试卷里还需要保存题目顺序和该考试下的分值，所以不能只在 `exam` 或 `question` 单表里解决。

### Q9：自动组卷怎么做？

自动组卷一般按课程、题型、难度、数量等条件筛选题库，再随机选取满足条件的题目，写入 `exam_question`。如果题量不足，应该抛出业务异常，而不是生成不完整试卷。生成后还要更新考试总分。

### Q10：手动组卷和自动组卷有什么区别？

手动组卷由教师主动勾选题目，适合精确控制试卷内容；自动组卷按条件和数量随机抽题，适合快速生成试卷。两者最终都落到 `exam_question` 表，只是选题来源不同。

### Q11：考试发布后为什么要锁定试卷？

因为学生开始答题后，题目和分值如果被修改，会导致成绩不可追溯。例如学生答的是旧题，但教师看到的是新题。发布后锁定试卷可以保证考试数据一致性。

### Q12：如何限制学生重复参加同一场考试？

`student_exam` 表有 `student_id + exam_id` 的唯一约束，表示某个学生对某场考试只有一条考试记录。Service 层也会判断当前考试状态，防止重复开始或重复提交。

### Q13：如何处理重复提交？

提交接口要先查询学生考试状态。如果已经提交，就返回已有结果或提示不能重复提交；如果仍在进行中，才允许写入答案并计算分数。数据库唯一约束也能防止同一题重复写入多条答案。

### Q14：客观题自动判分怎么做？

客观题提交后，后端取学生答案与题库标准答案比较，正确则给题目分值，错误则给 0 分，并写入 `is_correct` 和 `score`。多选题要注意答案顺序和集合比较，不能只做普通字符串比较。

### Q15：主观题为什么不能直接自动判分？

主观题通常没有唯一标准答案，需要教师根据语义、步骤和表达评分。因此系统把主观题答案标记为待批改，教师评分后再更新分数和总分。AI 可以辅助，但不应在这个项目里夸大为完全自动判分。

### Q16：错题记录什么时候生成？

一般在提交判分后生成。客观题答错可以直接写入错题；主观题要等教师批改后，如果得分未满或判定错误，再写入错题。错题记录保存学生答案和正确答案，便于复习。

### Q17：Excel 题库导入如何校验？

需要校验课程是否存在、题型是否合法、题干是否为空、分值是否大于 0、选项格式是否符合题型要求、答案是否符合选项范围。同时要检测文件内重复和数据库已有重复，错误要定位到具体行，便于教师修改。

### Q18：为什么 Excel 导入要返回错误行，而不是直接失败？

批量导入经常会有部分数据不规范。如果只返回“导入失败”，用户不知道哪里错了。返回具体行号和错误原因，能提高可用性，也体现后端对异常处理和用户体验的考虑。

### Q19：异常行为记录怎么实现？

前端监听切换页面、窗口失焦、退出全屏、复制、粘贴、右键等行为，调用后端接口上报。后端把异常写入 `exam_violation`，教师可以按考试或学生查看异常记录。系统只做风险提示，不建议单次异常直接判定结果。

### Q20：成绩统计怎么做？

成绩统计主要基于 `student_exam.total_score` 和 `student_answer`。可以统计平均分、最高分、最低分、分数段分布、题目正确率和学生错题。前端使用 ECharts 展示图表。

### Q21：RAG 知识库怎么设计？

知识库文档表保存文件元数据，分片表保存每个 chunk 的文本内容和可选 embedding。提问时按课程检索相关片段，再把问题和片段拼成 prompt 交给模型。这个设计是轻量级的，适合课程项目和演示。

### Q22：为什么不用 Milvus 或 Elasticsearch？

当前项目是课程设计和简历项目，数据规模有限，用 MySQL 存 chunk 和 embedding 信息更容易部署和演示。后续如果文档量很大，可以把检索层替换成 pgvector、Milvus 或 Elasticsearch。

### Q23：项目如何处理跨域？

开发环境前端可以通过 Vite 代理 `/api` 请求到后端，后端也可以配置允许来源。生产环境应限制允许的前端域名，不建议直接放开所有来源。

### Q24：统一返回格式有什么好处？

统一返回格式可以让前端稳定处理成功、失败、鉴权过期和业务错误。比如统一包含 `code`、`message`、`data`，前端拦截器就可以集中处理 401、403 和业务异常。

### Q25：项目启动失败你怎么排查？

先看后端日志和端口占用，再检查 MySQL 是否启动、数据库是否初始化、密码环境变量是否正确。前端问题则检查 Node 版本、依赖安装、Vite 代理和接口地址。中文乱码要检查数据库、连接串和 SQL 导入方式是否统一使用 utf8mb4。

### Q26：如果题库数据量变大，查询如何优化？

可以给课程、题型、难度、知识点等常用筛选字段建立组合索引；分页查询避免一次性查全量；题干搜索可以考虑全文索引或搜索引擎；热门统计可以做缓存或异步汇总。

### Q27：如果多人同时提交考试，会有什么问题？

主要关注重复提交、答案重复写入、总分重复计算和事务一致性。解决方式是状态校验、唯一约束、事务控制和幂等处理。对于大规模同时提交，还要考虑数据库连接池和写入压力。

### Q28：这个项目还有哪些可以继续优化？

可以增加更完整的自动化测试、试卷版本号、批改记录审计、错题复习计划、题目标签多对多、缓存成绩统计、RAG 接入向量数据库，以及更完善的前端路由权限控制。

## 6. 面试时不要踩的坑

- 不要说“精通 Spring Security”，应该说“项目中使用 Spring Security + JWT 实现认证授权”。
- 不要说“高并发系统”，可以说“通过唯一约束、状态校验和事务意识处理重复提交等问题”。
- 不要说“AI 自动完全判卷”，可以说“AI 出题和 RAG 答疑是扩展能力，核心考试流程不依赖 AI”。
- 不要把前端隐藏按钮当成权限控制，必须强调后端接口和 Service 层校验。
- 不要只背功能，要能讲清楚表关系、状态流转和异常处理。
