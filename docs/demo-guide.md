# 在线考试系统演示指南

## 1. 项目定位

本项目是一个基于 Spring Boot + Vue 3 的前后端分离在线考试系统，面向管理员、教师、学生三类用户。系统覆盖用户管理、课程管理、题库维护、Excel 批量导入、自动组卷、手动组卷、试卷预览、考试发布、学生在线答题、客观题自动判分、主观题人工批改、成绩统计、错题记录和考试异常行为监控等流程。

这个项目适合在简历和面试中作为 Java 后端项目展示：既能讲清楚权限认证、数据库设计、业务状态流转，也能通过前端页面演示完整业务闭环。

## 3. 推荐演示流程

### 步骤一：登录系统

使用教师账号 `teacher / 123456` 登录系统，先展示登录页、工作台和左侧菜单。重点说明系统通过 Spring Security + JWT 完成登录认证，通过角色控制前端菜单和后端接口访问。

### 步骤二：维护题库

进入题库管理页面，展示题目列表、课程筛选、题型、难度、知识点和关键词查询。可以重点演示 Excel 模板下载、批量导入、导出，以及导入失败时的错误行提示。

### 步骤三：创建考试并组卷

进入考试管理页面，创建考试后进入自动组卷或手动组卷。自动组卷适合展示按题型、数量和难度比例抽题；手动组卷适合展示按课程筛选题目、调整题目顺序和分值，并通过试卷预览确认最终内容。发布考试后，系统会禁止继续修改试卷，保证考试公平性。

### 步骤四：学生在线答题

切换到学生账号 `student / 123456`，进入可参加考试列表，开始考试。展示倒计时、不同题型作答、异常行为上报、提交试卷，以及系统如何防止重复提交。

### 步骤五：教师批改与成绩统计

切回教师账号，进入主观题批改页面，为简答题评分并填写评语。随后进入统计分析页面，展示参加人数、平均分、最高分、最低分、成绩分布和题目正确率图表。

### 步骤六：异常行为监控

进入考试监控页面，展示学生切换页面、窗口失焦、退出全屏、复制、粘贴、右键等异常行为记录。说明当前实现是记录和统计异常，不会自动强制交卷，便于教师综合判断。

## 4. 面试讲解重点

### 为什么使用 Spring Security + JWT？

Spring Security 负责认证入口、接口鉴权、异常处理和方法级权限控制，JWT 负责前后端分离场景下的无状态登录。登录成功后服务端签发 token，前端后续请求携带 `Authorization: Bearer ...`，后端过滤器解析 token 并写入安全上下文。

### 如何设计三角色权限？

系统角色包括 ADMIN、TEACHER、STUDENT。后端通过 `@PreAuthorize` 控制接口访问，例如用户管理仅管理员可访问，题库、组卷、批改由教师或管理员访问，在线考试、错题本和学生统计仅学生访问。前端路由也配置了角色元信息，避免用户看到无权限菜单。

### 如何避免教师越权访问其他教师的数据？

课程创建时教师角色会自动绑定当前登录用户 ID，部分业务会根据当前用户 ID 校验数据归属。例如教师不能管理不属于自己的课程，学生只能查看自己的考试记录和错题记录。真实生产环境中还可以继续把教师维度的数据隔离补充到考试、统计、批改等查询条件中。

### 自动组卷和手动组卷如何设计？

自动组卷根据考试课程、题型、题目数量、难度等条件从题库中抽题，题库不足时返回明确错误。手动组卷则先加载当前课程下可选题目，教师选择题目后保存题目顺序和分值，系统重新计算试卷总分。

### Excel 导入如何做校验和错误提示？

Excel 导入使用 Apache POI 解析文件，将每一行转换为导入 DTO 后逐行校验课程、题型、答案、分值、难度和题干内容。系统会检测数据库中已有重复题目，也会检测 Excel 文件内部重复题目，最终返回成功数量和错误行信息，便于用户修正。

### 主观题和客观题判分流程如何区分？

单选、多选、判断和填空题可以在学生提交后按标准答案自动判分。简答题没有唯一标准答案，提交后进入待批改状态，由教师人工评分并填写评语，批改完成后系统重新计算最终成绩。

### 考试发布后为什么要锁定试卷？

发布后的试卷已经面向学生开放，如果继续修改题目、分值或顺序，会影响已答题学生和未答题学生之间的公平性。系统通过业务状态判断，禁止对非草稿状态的试卷继续修改。

### Redis、Docker、CI 或 AI/RAG 功能如何作为加分项介绍？

当前仓库已包含 Docker Compose、Redis、GitHub Actions CI、AI 出题和 RAG 课程知识库相关能力。面试中可以把 Docker Compose 作为“一键演示环境”，把 CI 作为“提交后自动验证后端测试和前端构建”，把 AI/RAG 作为“题目生成、资料解析、课程知识库答疑”的项目亮点。Redis 可结合实时监考、缓存或限流继续展开。

## 5. 常见面试问题

### 1. JWT 登录流程是什么？

用户提交用户名和密码，后端用 BCrypt 校验密码；校验成功后生成 JWT，前端保存 token。后续请求携带 token，后端过滤器解析后把用户信息放入 Spring Security 上下文。

### 2. Spring Security 如何做接口鉴权？

系统在安全配置中放行登录和注册接口，其余接口都要求登录。具体角色权限通过 `@PreAuthorize` 标注在 Controller 上，例如 `hasRole('ADMIN')` 或 `hasAnyRole('ADMIN','TEACHER')`。

### 3. BCrypt 为什么比明文密码安全？

BCrypt 会对密码做带盐哈希，即使相同密码也可能得到不同密文，数据库泄露时攻击者不能直接看到用户明文密码。它还可以通过计算成本参数提高暴力破解成本。

### 4. 题库 Excel 导入如何避免重复题？

导入时会按课程、题干等信息检查数据库已有题目，同时也会检查 Excel 文件内部是否出现重复题目。重复项不会直接写入，并会返回错误行提示。

### 5. 自动组卷的核心筛选条件有哪些？

核心条件包括考试所属课程、题型、题目数量和难度要求。系统只从当前课程题库中选题，避免把其他课程题目混入试卷。

### 6. 手动组卷如何保证题目合法？

保存手动试卷时会校验题目 ID 是否存在、是否属于当前考试课程、题目是否重复、排序是否重复、分值是否大于 0。校验通过后才写入考试题目关系表。

### 7. 考试提交后如何计算成绩？

提交后系统逐题保存学生答案，客观题直接和标准答案比对并计算得分。包含主观题时成绩需要等待教师批改，批改完成后再汇总最终得分。

### 8. 主观题为什么不能直接自动判分？

简答题通常存在表达差异和开放性答案，简单字符串比对容易误判。当前项目采用教师人工批改，更符合在线考试系统的基础业务逻辑。

### 9. 错题记录什么时候生成？

学生提交考试后，系统会根据判分结果记录答错的客观题，便于学生后续查看错题和解析。公考专项练习中也会记录错题、错误次数和掌握状态。

### 10. 如何处理跨域？

后端提供 CORS 配置，前端开发环境通过 Vite 将 `/api` 请求代理到 `http://localhost:8080`。这样本地开发时前端访问后端接口更稳定。

### 11. 前后端如何约定接口返回格式？

后端使用统一响应对象 `Result`，包含状态码、消息和数据。前端 Axios 拦截器统一判断 `code`，成功时返回 `data`，失败时显示错误消息。

### 12. 项目如何初始化数据库？

先执行 `schema.sql` 创建数据库和业务表，再执行 `data.sql` 写入演示账号、课程、题目和考试数据。后续功能迁移脚本包括批改、异常监控和公考专项练习相关表。

### 13. 为什么要写 Docker Compose？

Docker Compose 可以把 MySQL、Redis、后端和前端等依赖统一编排，降低面试官或同学本地运行项目的成本。本仓库已提供开发和生产 Compose 配置，演示时可以先复制 `.env.example`，再执行 `docker compose up -d --build`。

### 14. 项目启动失败你会如何排查？

先看端口是否被占用，再检查 MySQL 是否启动、数据库是否初始化、`MYSQL_PASSWORD` 是否正确。后端再看 Maven 编译和启动日志，前端看 npm 依赖安装、Vite 代理和接口请求错误。

### 15. 如果数据量变大，题库查询如何优化？

可以给课程 ID、题型、难度、知识点、创建人和创建时间建立合适索引，分页查询避免一次加载过多数据。关键词搜索可以考虑全文索引或接入搜索引擎。

### 16. 如果多人同时考试，哪些地方容易出问题？

容易出问题的点包括重复提交、考试记录并发创建、主观题批改状态更新、统计实时计算压力和数据库连接数。可以用唯一约束、事务、幂等校验、连接池调优和缓存来增强稳定性。

### 17. 考试异常行为监控是怎么做的？

前端监听切屏、失焦、退出全屏、复制、粘贴、右键等行为并上报后端。后端记录异常类型和时间，并做短时间重复上报限制，教师端按学生维度查看风险等级。

### 18. 项目里哪个点最适合体现后端能力？

比较适合讲题库 Excel 导入、组卷校验、JWT 鉴权、发布后锁定、提交判分和批改统计。这些功能既有业务规则，也涉及数据库、事务、权限和异常处理。
